===========================================================================
  SDE_Spark — Experimental Evaluation: Preliminary Results
  Experiment B: Throughput vs Ingestion Rate
  Date: 2026-03-26
===========================================================================

1. OBJECTIVE
------------
Evaluate whether the SDE_Spark pipeline can sustain increasing data
ingestion rates without falling behind — the same experiment performed
by Kontaxakis (2020, Figure 24) on the original SDE_Flink implementation.


2. TEST SETUP
-------------
Hardware:
  - CPU:    AMD Ryzen 7 5800H (8 cores / 16 threads)
  - RAM:    16 GB
  - Disk:   512 GB NVMe SSD
  - OS:     Windows 11

Software:
  - Apache Spark 3.5.3 (Structured Streaming, local[4] mode)
  - Apache Kafka 7.5.3 (single broker, Docker)
  - Java 17, Kryo serialization

Pipeline Configuration:
  - 4 shuffle partitions
  - 2-second trigger interval
  - 2 GB driver memory
  - Spark UI disabled (to reduce overhead)

Experiment Parameters:
  - Streams:     10 distinct streamIDs (currency pairs)
  - Synopses:    1 CountMin Sketch (epsilon=0.002, delta=0.01)
  - Parallelism: noOfP=1 (GREEN path — single partition)
  - Stock pool:  50 stock symbols with random prices [50-1000]
  - Duration:    60 seconds per rate (+ 10 second warmup)
  - Estimates:   1 ESTIMATE request every 30 seconds


3. METHODOLOGY
--------------
A custom LoadTestProducer generates synthetic stock market data and
sends it to Kafka at a controlled, steady rate. The data follows the
same JSON schema as the original SDE_Flink implementation:

  {"dataSetkey":"Forex","streamID":"EURUSD",
   "values":{"StockID":"AAPL","price":"185.50"}}

The SDE_Spark pipeline reads from Kafka, routes data through the
6-layer architecture (Ingestion -> Routing -> Synopsis Processing ->
Path Splitting -> Aggregation -> Output), and writes estimations
back to Kafka.

A StreamingQueryListener (ThroughputListener) inside the pipeline
captures per-batch metrics: number of input rows, input rate,
processed rate, and batch duration. These are written to CSV for
analysis.

Three runs were executed sequentially, each at a different target
ingestion rate: 100, 500, and 1000 messages per second.

5. KEY FINDINGS
---------------
a) LINEAR THROUGHPUT SCALING
   The pipeline sustains all tested ingestion rates. When the input
   rate increases from 100 to 1,000 msg/sec (10x), the processed
   rate increases proportionally (86 to 960 rows/sec, ~11x).
   At no point did the pipeline fall behind.

b) CONSTANT BATCH DURATION
   Average batch duration remains ~6.5 seconds regardless of the
   ingestion rate. This indicates the bottleneck is per-batch
   overhead (state serialization, Kafka polling, task scheduling),
   not synopsis computation. Higher data volumes per batch are
   processed with negligible additional cost.

c) SYNOPSIS CORRECTNESS
   The CountMin sketch correctly accumulated price values and
   returned a valid estimation. No data corruption or parsing
   errors were observed during the measurement phase.

d) COMPARISON WITH KONTAXAKIS (2020)
   These results mirror the pattern shown in Figure 24 of the
   original thesis: near-linear throughput scaling with increasing
   ingestion rate. The SDE_Spark implementation preserves the
   scalability characteristics of the Flink-based SDE.


6. LIMITATIONS
--------------
- Single-node execution: all Spark executors share the same CPU
  and memory. Multi-node cluster experiments are needed to evaluate
  horizontal scalability (comparable to Kontaxakis Figure 23).

- Limited stream count (10 vs 50/500/5000 in Kontaxakis) due to
  16 GB RAM constraint on test hardware.

- Only GREEN path tested (noOfP=1). PURPLE path (parallel synopsis
  processing with aggregation) requires separate evaluation.

- No comparison against SDE_Flink on identical hardware. A direct
  Spark vs Flink comparison would be a significant contribution.
